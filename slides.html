What is JavaScript?

JS runs in two places

  • Browser — manipulates UI, handles user events
  • Node.js — server-side, scripts, build tools

Key facts

  • Interpreted, not compiled
  • Single-threaded + event loop
  • Dynamically typed
Think of Node.js like your Go/Java backend. The browser JS is the "client SDK" of your app.

Variables & Types


// Avoid var — it has weird scoping

// let — can be reassigned

count = 1

// const — can't be reassigned (prefer this by default)

// Types: string, number, boolean, null, undefined, object, array

Functions & Arrow Functions


// Classic function
function greet(name) {
  return `Hello, ${name}!`
}

// Arrow function (modern, preferred)
const greet = (name) => `Hello, ${name}!`

// Arrow with body
const add = (a, b) => {
  const result = a + b
  return result
}

console.log(greet("Backend Dev")) // Hello, Backend Dev!

Array Methods You'll See Everywhere


const users = [
  { name: "Alice", active: true  },
  { name: "Bob",   active: false },
  { name: "Carol", active: true  },
]

// map — transform each item (returns new array)
const names = users.map(u => u.name)
// ["Alice", "Bob", "Carol"]

// filter — keep items matching condition
const active = users.filter(u => u.active)
// [{ name: "Alice" }, { name: "Carol" }]

// reduce — collapse to single value
const count = users.reduce((acc, u) => acc + (u.active ? 1 : 0), 0)
// 2
These are like SELECT, WHERE, and aggregations in SQL.

Async / Await


// Callbacks (old, avoid)
fetchUser(id, function(err, user) { ... })

// Promises (intermediate)
fetchUser(id)
  .then(user => console.log(user))
  .catch(err => console.error(err))

// async/await (modern, readable)
async function loadUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`)
    const user = await response.json()
    console.log(user)
  } catch (err) {
    console.error("Failed:", err)
  }
}
Same concept as goroutines/channels or async handlers in Go — don't block the thread.

What is the DOM?

The browser parses HTML and creates a tree of objects — the Document Object Model.


<div id="app">
  <h1>Hello</h1>
  <p>World</p>
</div>
  

// Select elements
const heading = document.querySelector("h1")
const app     = document.getElementById("app")

// Modify them
heading.textContent = "Updated!"
app.style.background = "lightblue"

// Create new elements
const btn = document.createElement("button")
btn.textContent = "New Button"
app.appendChild(btn)
The DOM is like an in-memory JSON tree that represents the page.

Event Handling


const btn = document.querySelector("#submit-btn")

btn.addEventListener("click", (event) => {

  console.log("Button clicked!", event.target)
})

// Common events
// click, submit, keydown, keyup, change, input, focus, blur
Like HTTP middleware or webhook handlers — "when X happens, run Y".

How Browsers Render Pages

1. Parse HTML → build DOM tree
2. Parse CSS → build CSSOM
3. Combine → Render tree
4. Layout → calculate positions
5. Paint → pixels on screen

Why it matters: changing the DOM triggers re-layout. This is why React batches DOM updates.

Modern JS & ES6+ Patterns

Destructuring


// Object destructuring
const user = { name: "Alice", age: 30, role: "admin" }

const { name, role } = user
console.log(name) // "Alice"

// With rename
const { name: userName } = user

// Default value
const { theme = "light" } = user

// Array destructuring
const [first, second, ...rest] = [1, 2, 3, 4, 5]
console.log(first) // 1
console.log(rest)  // [3, 4, 5]
Like unpacking a struct in Go or pattern matching in Rust.

Spread & Rest


// Spread — expand an array/object
const defaults = { theme: "light", lang: "en" }
const userPrefs = { lang: "de", fontSize: 14 }

const config = { ...defaults, ...userPrefs }
// { theme: "light", lang: "de", fontSize: 14 }

// Spread array
const a = [1, 2, 3]
const b = [...a, 4, 5]  // [1, 2, 3, 4, 5]

// Rest — collect remaining args
function log(first, ...others) {
  console.log(first)  // "hello"
  console.log(others) // ["world", "!"]
}
log("hello", "world", "!")

Modules: import / export


// utils.js — named exports
export const PI = 3.14
export function double(n) { return n * 2 }

// api.js — default export
export default class ApiClient { ... }

// main.js — importing
import ApiClient from "./api.js"
import { PI, double } from "./utils.js"

// Import everything
import * as utils from "./utils.js"
utils.double(5) // 10
Same concept as Go packages / Java imports — each file is its own module/namespace.

Closures & Scope


// A closure "closes over" outer variables
function makeCounter() {
  let count = 0          // private to makeCounter

  return {
    increment: () => ++count,
    decrement: () => --count,
    value:     () => count,
  }
}

const counter = makeCounter()
counter.increment() // 1
counter.increment() // 2
counter.value()     // 2

You'll see this pattern inside React hooks constantly.

Like an object with private fields — encapsulated state.

this — Just Enough


// In a method — `this` = the object
const obj = {
  name: "EJOT",
  greet() { console.log(this.name) } // "EJOT" ✅
}

// Arrow functions DON'T have their own `this`
const obj2 = {
  name: "EJOT",
  greet: () => console.log(this.name) // undefined ❌
}

// React rule of thumb:
// Use arrow functions for event handlers → no this binding needed

Good news: with React function components + hooks, you almost never need this.

Why React Exists

Without React


// Every state change:
// 1. Update the data
// 2. Find the right DOM elements
// 3. Update them manually
// 4. Don't forget edge cases
// 5. Gets messy fast at scale
const el = document.getElementById("count")

// × No structure
// × Hard to reuse
// × Hard to test

With React


// Describe WHAT to show, not HOW to update
function Counter() {
  const [count, setCount] = useState(0)

  return (
    <button onClick={() => setCount(count + 1)}>
      Count: {count}
    </button>
  )
}
// ✓ Declarative
// ✓ Reusable component
// ✓ React handles DOM updates

Components

Everything in React is a component — a JS function that returns UI.


// A component is just a function
function UserCard({ name, email }) {
  return (
    <div className="card">
      <h2>{name}</h2>
      <p>{email}</p>
    </div>
  )
}

// Compose components like LEGO
function App() {
  return (
    <div>
      <UserCard name="Alice" email="alice@ejot.com" />
      <UserCard name="Bob"   email="bob@ejot.com"   />
    </div>
  )
}
Components are like microservices for UI — small, reusable, composable.

JSX — Not Magic, Just Syntax Sugar

What you write (JSX)


const element = (
  <div className="box">
    <h1>Hello</h1>
  </div>
)

What it compiles to


const element = React.createElement(
  "div",
  { className: "box" },
  React.createElement("h1", null, "Hello")
)

Key JSX rules:

  • Use className not class
  • All tags must be closed: <br />
  • One root element (or use <>...</> Fragment)
  • JS expressions go in { }

Props — Passing Data Down


// Parent passes data via props (like function arguments)
function App() {
  return <Greeting name="Alice" isAdmin={true} />
}

// Child receives props as an object
function Greeting({ name, isAdmin }) {
  return (
    <div>
      <p>Hello, {name}!</p>
      {isAdmin && <p>You have admin access.</p>}
    </div>
  )
}
Props = function parameters. They flow one way: parent → child (never the other way).

State — useState


import { useState } from "react"

function Counter() {
  // [currentValue, setterFunction] = useState(initialValue)
  const [count, setCount] = useState(0)

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)}>-</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  )
}

// When setCount() is called → React re-renders the component
State is like a local variable that, when changed, automatically triggers a UI refresh.

useEffect — Side Effects


import { useState, useEffect } from "react"

function UserProfile({ userId }) {
  const [user, setUser] = useState(null)

  useEffect(() => {
    // Runs AFTER the component renders
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => setUser(data))
  }, [userId]) // dependency array — re-run when userId changes

  if (!user) return <p>Loading...</p>
  return <p>{user.name}</p>
}
[] — run once on mount
[id] — run when id changes
Like OnInit / lifecycle hooks — run code when something mounts or changes.

Fetching Data from an API


function UserList() {
  const [users, setUsers]   = useState([])
  const [loading, setLoading] = useState(true)
  const [error, setError]   = useState(null)

  useEffect(() => {
    fetch("/api/users")
      .then(res => {
        if (!res.ok) throw new Error("Failed to fetch")
        return res.json()
      })
      .then(data => setUsers(data))
      .catch(err => setError(err.message))
      .finally(() => setLoading(false))
  }, [])

  if (loading) return <p>Loading...</p>
  if (error)   return <p>Error: {error}</p>
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>
}

Lifting State Up


// When two siblings need the same state → lift to parent
function App() {
  const [search, setSearch] = useState("")

  return (
    <div>
      <SearchBar value={search} onChange={setSearch} />
      <UserList  filter={search} />
    </div>
  )
}

// Child receives state + setter as props
function SearchBar({ value, onChange }) {
  return <input value={value} onChange={e => onChange(e.target.value)} />
}
Like sharing a variable between two services via a shared database or message bus.

Routing with React Router


import { BrowserRouter, Routes, Route, Link } from "react-router-dom"

function App() {

Why TypeScript?

JavaScript


function getUser(id) {
  return fetch(`/api/users/${id}`)
}

// What does this return?
// What type is `id`?
// No idea until runtime 😬

TypeScript


interface User {
  id: number
  name: string
  email: string
}

async function getUser(id: number): Promise<User> {
  const res = await fetch(`/api/users/${id}`)
  return res.json()
}
// ✓ IDE autocomplete
// ✓ Compile-time errors
TypeScript is to JavaScript what Go's type system is to dynamically typed languages — same idea, compile-time safety.

Basic Types


// Primitives
const name: string  = "Alice"
const age:  number  = 30
const active: boolean = true

// Arrays
const ids:   number[] = [1, 2, 3]
const names: string[] = ["Alice", "Bob"]

// Object type inline
const user: { name: string; age: number } = { name: "Alice", age: 30 }

// Union — can be one of several types
let id: string | number = 42
id = "abc"  // also valid

// Optional — may or may not exist
const nickname?: string  // undefined is fine

Interfaces & Type Aliases


// Interface — for object shapes (preferred for APIs)
interface User {
  id:       number
  name:     string
  email:    string
  role?:    "admin" | "viewer"  // optional + literal union
}

// Type alias — more flexible, can alias primitives too
type UserId = number
type Status = "active" | "inactive" | "pending"

// Extending an interface
interface AdminUser extends User {
  permissions: string[]
}

// Using them
function renderUser(user: User) {
  return `${user.name} <${user.email}>`
}
Interfaces = Go structs / Java interfaces. Type aliases = type definitions.

Typing Functions & Generics


// Typed function
function add(a: number, b: number): number {
  return a + b
}

// Arrow function
const greet = (name: string): string => `Hello, ${name}!`

// Generics — reusable with any type
function first<T>(arr: T[]): T | undefined {
  return arr[0]
}

first([1, 2, 3])        // type: number
first(["a", "b", "c"]) // type: string

// Common in React (you'll read this a lot)
const [user, setUser] = useState<User | null>(null)
Generics in TS = generics in Go — write once, work with any type.

React Router — Quick Recap


import { BrowserRouter, Routes, Route, Link } from "react-router-dom"

function App() {
  return (
    <BrowserRouter>
      <nav>
        <Link to="/">Home</Link>
        <Link to="/users">Users</Link>
      </nav>
      <Routes>
        <Route path="/"           element={<Home />}       />
        <Route path="/users"      element={<UserList />}   />
        <Route path="/users/:id"  element={<UserDetail />} />
      </Routes>
    </BrowserRouter>
  )
}
Routes ≈ HTTP path handlers in your API router. :id is a URL path parameter.

useParams & useNavigate


import { useParams, useNavigate } from "react-router-dom"

// useParams — read the :id from the URL
function UserDetail() {
  const { id } = useParams<{ id: string }>()  // /users/42 → id = "42"
  return <h1>User #{id}</h1>
}

// useNavigate — redirect programmatically
function LoginForm() {
  const navigate = useNavigate()

  const handleLogin = async () => {
    await authApi.login(credentials)
    navigate("/dashboard")   // redirect after login
    navigate(-1)             // go back (like browser back button)
  }
}
useNavigate ≈ a server-side redirect — change the URL without a link click.

Protected Routes


import { Navigate } from "react-router-dom"

// Wrapper that checks auth before rendering the route
function ProtectedRoute({ children }: { children: React.ReactNode }) {
  const { user } = useAuthStore()

  if (!user) return <Navigate to="/login" replace />
  return <>{children}</>
}

// Usage
<Routes>
  <Route path="/login"     element={<Login />} />
  <Route path="/dashboard" element={
    <ProtectedRoute><Dashboard /></ProtectedRoute>
  } />
</Routes>
Like auth middleware in Go/Express — intercept the request, check the token, continue or redirect.

React Query — The Problem It Solves

Without React Query


// Boilerplate you write EVERY time
const [data, setData]     = useState(null)
const [loading, setLoading] = useState(true)
const [error, setError]   = useState(null)

useEffect(() => {
  setLoading(true)
  fetch("/api/users")
    .then(r => r.json())
    .then(setData)
    .catch(setError)
    .finally(() => setLoading(false))
}, [])
// + no caching
// + refetches on every mount

With React Query


import { useQuery } from "@tanstack/react-query"

function UserList() {
  const { data, isLoading, error } = useQuery({
    queryKey: ["users"],
    queryFn: () =>
      fetch("/api/users").then(r => r.json())
  })

  if (isLoading) return <p>Loading...</p>
  if (error)     return <p>Error!</p>
  return <ul>{data.map(u => <li key={u.id}>{u.name}</li>)}</ul>
}
// + automatic caching
// + background refetch
// + deduplication

React Query — useMutation


import { useMutation, useQueryClient } from "@tanstack/react-query"

function CreateUser() {
  const queryClient = useQueryClient()

  const mutation = useMutation({
    mutationFn: (newUser) =>
      fetch("/api/users", {
        method: "POST",
        body: JSON.stringify(newUser),
        headers: { "Content-Type": "application/json" },
      }).then(r => r.json()),

    onSuccess: () => {
      // Invalidate cache → triggers refetch of user list
      queryClient.invalidateQueries({ queryKey: ["users"] })
    },
  })

  return (
    <button onClick={() => mutation.mutate({ name: "Alice" })}>
      {mutation.isPending ? "Saving..." : "Add User"}
    </button>
  )
}
Query cache invalidation ≈ cache busting after a write operation in your backend.

Zustand — Global UI State

When to use Zustand vs useState?
Use useState for local, component-level state.
Use Zustand when multiple components across the tree need the same state. 

import { create } from "zustand"

interface AuthStore {
  user:    User | null
  token:   string | null
  login:   (user: User, token: string) => void
  logout:  () => void
}

const useAuthStore = create<AuthStore>((set) => ({
  user:  null,
  token: null,
  login:  (user, token) => set({ user, token }),
  logout: ()            => set({ user: null, token: null }),
}))

Zustand — Using the Store


// In any component — no Provider needed!
function Navbar() {
  const { user, logout } = useAuthStore()

  return (
    <nav>
      <span>Hello, {user?.name}</span>
      <button onClick={logout}>Logout</button>
    </nav>
  )
}

function LoginForm() {
  const login = useAuthStore((state) => state.login)

  const handleLogin = async () => {
    const { user, token } = await authApi.login(credentials)
    login(user, token) // updates store → Navbar re-renders
  }

  return <form onSubmit={handleLogin}>...</form>
}
Zustand store = a singleton service with reactive subscriptions — like a simple in-memory event bus.

React Query vs Zustand — When to Use What

React Query — for server state
Data that comes from an API: users, products, orders. It has a source of truth on the server.
Zustand — for client/UI state
Auth session, modal open/close, sidebar state, selected items. Lives only in the browser.
useState — for local component state
Form inputs, counters, toggles — state that only this component cares about.
